# xOS — brief

Share this first. More detail: [VISION.md](./VISION.md), then [README.md](./README.md).

---

xOS is a Linux desktop built around **what you are trying to do**, not around apps and browser tabs. Today, real work means jumping between editors, tickets, logs, docs, internal tools, and websites, re-finding context each time. xOS starts from your goal. If the system already knows how to do that kind of thing, it does it and shows the result in a focused workspace. If it does not, an agent helps you wire it up (with clear permissions), saves that skill, and next time you do not repeat the click path.

The shell is first-class: a **fast normal path** (CLI, terminal, launcher-class actions with no model wait) **and** a plain-language goal path into the agent stack (Archy/Enso-style *command nucleus*). When a pattern is clear, it is **saved as a sandboxed capability** so the next time is not freeform inventing. The desktop is Wayland-native with a **modal** focus (work mode + agent as center, not window soup or a ricing product). Agents are not one chat process with your full rights: each gets a **user account**, a **systemd-class unit** (isolation, restarts), ACLs, and durable memory/data paths. Soft stop can **checkpoint** live process state with CRIU and **snapshot** data with **btrfs**—not only kill the process. The base is radical and small when it can be: Linux plus a **BusyBox-class multicall core**, a real **system libc we stay close to for v1** (apps depend on libc details—malloc swaps need an app matrix), and an explicit story for **executable/library formats, linkers, loaders, and debug symbols** so agent processes are dump-friendly. This is **not** “just another Sway” and **not** “shell out to Grok on Ubuntu”: the product wedge is an **OS-level agent harness** (identities, capabilities, dual path, CRIU, budgets). **QEMU-first** bring-up is fine; real hardware for local LLM and drivers. Models: remote for hard work, local when it fits—not every layer rewritten at once.

This is aimed at people who do hard jobs on computers all day: engineering, ops, research, design, writing. Version 1 will not cover every workflow. It only needs to prove the idea on a few real tasks: actual coding with editor and terminal, a real investigation or write-up with sources, and adding one non-trivial skill then using it again. A pretty Linux image with a chat box and a few toy demos is not enough. When an agent acts, you should see what it is doing, stop it, and approve anything risky.
